# HireInsight Agent 中期答辩技术白皮书

> 面向中期检查答辩的结构化技术梳理，侧重系统框架而非代码细节。

---

## 一、项目定位

HireInsight Agent 是一个**面向校招求职场景的智能辅助平台**，核心理念是"用 AI 帮学生更好地看清市场、看清差距、看清自己"。系统围绕三个业务模块展开：

| 模块 | 解决的问题 |
|------|-----------|
| 📊 数据大屏 | 多源岗位数据汇聚、清洗、可视化分析，帮用户了解市场行情 |
| 🎤 面试模拟 | 简历解析 → 市场对比 → 差距诊断 → 定制化面试题生成 |
| 🧭 灯塔计划 | 面向低年级学生的技术倾向测评 + AI 学习路线规划 |

## 三、数据来源与数据管线

### 3.1 数据来源

岗位数据来自两大真实企业招聘官网的公开接口：

- **网易招聘**（hr.163.com）：通过分析其内部 API（`/api/hr163/position/queryPage`）直接获取结构化 JSON 数据
- **腾讯招聘**（careers.tencent.com）：使用 Playwright 无头浏览器拦截 API 响应（`/tencentcareer/api/post/Query`）获取结构化岗位数据

> 说明：网易招聘 API 为公开 RESTful 接口无需认证；腾讯招聘 API 需要浏览器 Cookie 认证，通过 Playwright 自动化浏览器拦截实现。

### 3.2 数据管线设计

```
爬虫获取原始 JSON → 多源格式统一转换 → 正则清洗标准化 → SQLite Upsert 入库
```

这条管线有三个关键设计：

1. **多源适配层**：网易和腾讯的原始数据结构不同（网易为 RESTful POST JSON，腾讯为 Playwright 拦截 API 响应），系统使用策略模式为每个来源维护独立的转换器，输出为统一的 20 字段标准中间格式
2. **清洗标准化**：将所有薪资格式（k/元/日薪/年薪）统一归算为"千元/月"，城市名/学历名做模糊匹配对齐，经验要求做结构化提取
3. **Upsert 去重**：基于 `(original_id, source)` 联合唯一索引，重复岗位自动更新而非新增

## 四、数据库底层设计

系统使用 **SQLite** 作为唯一数据库，轻量但设计规范，四张核心表如下：

| 表名 | 职责 | 关键设计 |
|------|------|---------|
| `users` | 用户账户体系 | PBKDF2-SHA256 哈希密码（10 万次迭代）+ 随机 Salt，支持头像 Base64 存储 |
| `job_positions` | 岗位数据仓库 | 30 列完整字段，联合唯一索引防重复，支持多维度组合条件查询 |
| `interview_history` | 面试模拟记录 | 复杂字段（面试题列表、报告内容）以 JSON 序列化存储 |
| `lighthouse_history` | 灯塔计划记录 | 用户答题数据和 AI 评估结果以 JSON 存储，按 `user_id` 隔离 |

**设计原则**：
- 用户数据严格隔离：所有历史记录查询和删除操作均校验 `user_id`，防止越权
- 参数化查询：所有 SQL 使用参数占位符，防止注入风险
- 自动列迁移：`users` 表支持后续新增字段自动补列，保证版本兼容

## 五、智能体融合机制

这是项目的核心技术亮点。系统使用 **LangGraph** 框架编排多 Agent 协作。

### 5.1 面试模拟工作流（三 Agent 串联）

```
[开始] → Market_Agent → Critic_Agent → Interviewer_Agent → [结束]
```

| Agent | 输入 | 输出 | 依赖 |
|-------|------|------|------|
| **Market Agent** | 用户目标岗位 + 公司 | 《市场趋势与竞争分析报告》 | 岗位数据库统计数据 |
| **Critic Agent** | 用户简历 + 市场报告 | 《技能差距诊断报告》（🟢🟡🟠🔴 四色等级） | RAG 检索的企业面经 |
| **Interviewer Agent** | 差距诊断结果 + 面经内容 | 3-5 道定制化面试题 | 前两个 Agent 的输出 |

**设计巧思**：这不是简单的"三个 Prompt 串行调用"，而是通过 LangGraph 的 StateGraph 将状态在节点间传递，前一个 Agent 的输出直接成为后一个 Agent 的决策依据，形成真正的"链式推理"。

### 5.2 灯塔计划工作流（评估 + 规划）

```
[开始] → QuestionFilter → Assessment → [结束]
```

- **QuestionFilter 节点**：LLM 从 10 道预设情景多选题中根据用户目标和年级筛选 5-7 道
- **Assessment 节点**：根据答题结果评估六维度技术倾向（前端/后端/AI数据/测试运维/产品/客户端，0-100 分），并生成结构化的学习路线图 JSON

### 5.3 容错降级设计

考虑到 LLM 服务的不稳定性，系统做了两层兜底：

1. **LLM 输出解析**：DeepSeek Reasoner 模型会在输出中混入思考链（CoT）文本，系统设计了"json 代码块匹配 → 首尾花括号截取 → 返回 None"三层 Fallback 解析
2. **图执行降级**：LangGraph 图执行异常时，自动 fallback 为纯 Python 顺序调用，保证核心功能可用

## 六、RAG 检索增强生成

### 6.1 技术路线

系统使用 **scikit-learn 的 TF-IDF + 余弦相似度** 实现轻量级向量检索，替代了原计划的 ChromaDB。

**替代原因**：ChromaDB 依赖 `chroma-hnswlib` 原生扩展，在 Windows + Python 3.12 环境下需要 Visual C++ Build Tools 编译安装，环境部署门槛过高。TF-IDF 方案纯 Python 实现，零原生依赖，对环境零要求。

### 6.2 应用场景

RAG 在面试模拟模块中发挥作用：当 Critic Agent 需要诊断"你和大厂要求差在哪"时，系统从面经知识库中检索 Top-2 条相关企业面经片段作为额外上下文注入 LLM，让差距分析更有针对性。

### 6.3 知识库现状

当前使用 5 条 Mock 面经数据（拼多多等企业），使用 pickle 序列化 TF-IDF 向量矩阵 + JSON 存储文档文本，存储在本地 `data/rag_store/` 目录。

### 6.4 后续扩展方向

当面经数据量增长到数千条时，可平滑切换为 `faiss-cpu`（有 Windows 预编译 wheel），检索性能更好，接口层面无需改动。

## 七、前端组件与技术栈映射

### 7.1 整体框架

前端基于 **Streamlit** 构建，采用双向索引路由表实现 6 页面切换，页面间通过侧边栏 `radio` 和页面内按钮双向联动。

### 7.2 各页面核心组件

| 页面 | 使用的 Streamlit 组件 | 额外技术 |
|------|----------------------|---------|
| **首页** | `st.markdown` 自定义 HTML/CSS 卡片布局 | 系统状态面板（动态检测 API Key/RAG/数据库可用性） |
| **数据大屏** | `st.sidebar` 多维筛选器 + `st.plotly_chart` + `st.dataframe` | Plotly 可视化（薪资柱状图、城市横向条形图、饼图），`st.cache_data` 缓存查询结果 |
| **面试模拟** | `st.file_uploader`（PDF）+ `st.spinner` 进度提示 + `st.markdown` 渲染报告 | pdfplumber 简历解析，LangGraph 工作流驱动状态流转 |
| **灯塔计划** | `st.radio` 单选答题 + `st.plotly_chart` 雷达图 + `st.markdown` 路线图 | LLM 动态选题 + 六维度能力评估 |

| **系统设置** | 纯信息展示 | 环境变量读取展示 |

### 7.3 UI 设计风格

采用 **"Creamy Clean 奶油风"** 设计语言：温暖奶茶色背景 + 巧克力褐色按钮 + 大圆角卡片容器。全部通过 CSS 注入实现，无需额外 UI 库。

## 八、关键技术选型理由

| 技术选择 | 理由 |
|---------|------|
| **Streamlit 而非 Vue/React** | 纯 Python 即可构建 UI，无需前后端分离，开发效率极高，适合以算法和 AI 为核心的展示型应用 |
| **LangGraph 而非 LangChain 原生 Chain** | StateGraph 提供显式的状态传递和条件路由，比 Chain 的隐式传递更可控，节点职责清晰 |
| **DeepSeek Reasoner** | 具备思维链推理能力，在技能差距诊断和面试题生成这类需要深度推理的场景中表现更好 |
| **SQLite 而非 MySQL/PostgreSQL** | 本地化部署需求，单文件数据库免安装免配置，同时完全满足当前数据规模 |
| **TF-IDF 而非 ChromaDB** | Windows 环境下免 C++ 编译依赖，部署即用；当前数据量级下精度足够 |
| **Plotly 而非 Echarts** | 与 Streamlit 原生集成，支持交互式图表，Python 端生成 |

## 九、项目亮点总结

1. **多 Agent 协作的 AI 工作流**：不是简单的"调 API 问一个问题"，而是通过 LangGraph 编排三个不同职责的 Agent 形成链式推理，每个 Agent 专注于一个子任务，分工明确、可独立调试

2. **真实数据驱动的分析能力**：岗位数据来自网易和腾讯真实招聘接口，经过清洗标准化后入库，而非依赖 LLM 的记忆/幻觉生成

3. **RAG 增强的个性化诊断**：将企业面经知识库与用户简历做交叉对比，让 AI 的差距分析有据可依

4. **全链路容错降级**：从 LLM 输出解析的多层 Fallback，到 LangGraph 执行异常的纯 Python 回退，保证系统的鲁棒性

5. **低门槛部署**：Python 环境 + `pip install` 即用，不需要数据库安装、不需要 C++ 编译环境、不需要 Docker，充分考虑了 Windows 用户的实际使用场景

6. **完整的数据管线**：爬虫 → 转换 → 清洗 → 存储 → 统计 → 可视化，形成从数据采集到前端展示的完整闭环

## 十、答辩预判 Q&A

> 以下问题是根据评委可能提问方向整理的预判回答。

**Q1：你们的岗位数据哪来的，是不是随便写的？**
答：数据来自网易和腾讯的招聘官网公开接口（网易为 RESTful API，腾讯通过 Playwright 自动浏览器拦截）。每个岗位都有 `original_id` 和 `source` 字段溯源，数据大屏中每条记录都附带官网链接，可点击跳转验证。

**Q2：SQLite 能撑住吗？为什么不选 MySQL？**
答：当前岗位数据量在数千到万级别，SQLite 的并发读写完全够用。而且是本地化部署方案，免安装免配置，符合我们"让用户开箱即用"的设计理念。如果未来要做 SaaS 多租户，迁移到 MySQL/PostgreSQL 只需要改连接层，数据层接口已做了抽象。

**Q3：三个 Agent 是怎么串起来的？和一次调三个 Prompt 有什么区别？**
答：我们用的是 LangGraph 的 StateGraph，本质上是一个有向无环图（DAG）。和前一个 Agent 的输出通过共享状态自动流转到下一个 Agent。关键区别在于：三个 Prompt 是"各问各的、互不依赖"，而我们的 Market → Critic → Interviewer 是"后一个的决策依赖前一个的结果"，形成了真正的链式推理。

**Q4：RAG 为什么用 TF-IDF，是不是太简单了？**
答：选 TF-IDF 是经过权衡的工程决策。原计划用 ChromaDB 但 Windows 下安装需要 C++ 编译环境，对普通用户不友好。TF-IDF 在当前 5 条面经数据下精度足够，且纯 Python 零依赖，开箱即用。后续数据量增长时可以平滑切换到 faiss。

**Q5：如果 DeepSeek API 挂了，系统会怎样？**
答：我们做了多层降级。LLM 输出解析有三层 Fallback；LangGraph 图执行有纯 Python 顺序调用的回退方案。最差情况下用户会看到友好的错误提示，但不会白屏崩溃。

**Q6：灯塔计划的雷达图六维度是你们拍脑袋定的吗？**
答：六个维度（前端/后端/AI数据/测试运维/产品/客户端）覆盖了校招技术岗的主要方向。维度定义在题库设计中已预设，但具体打分由 LLM 根据用户答题结果动态计算，不是固定的。

**Q7：用户密码怎么存的，安全吗？**
答：使用 PBKDF2-SHA256 算法，10 万次迭代加随机 Salt，即使数据库文件泄露也无法反向破解密码。
