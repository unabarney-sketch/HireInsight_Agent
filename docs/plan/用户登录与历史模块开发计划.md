# 用户登录与历史模块开发计划

## 一、项目概述

为 HireInsight-Agent 添加标准多用户登录系统、SQLite 持久化对话历史（面试模拟 + 灯塔计划）、历史记录查看功能，并将前端全局 CSS 全面重构为 **高级柔和奶油风（Creamy Clean Style）**。

## 二、需求分析

### 2.1 功能需求

| 编号 | 需求 | 优先级 |
|------|------|--------|
| R1 | 用户注册/登录：标准账号密码模式，密码 PBKDF2 哈希存储 | 高 |
| R2 | 多用户数据隔离：每用户只能查看自己的历史记录 | 高 |
| R3 | 面试模拟历史持久化：每次分析自动保存，提供"历史对话"按钮 | 高 |
| R4 | 灯塔计划历史持久化：每次测评自动保存，提供"历史记录"按钮 | 高 |
| R5 | 历史记录管理：查看列表、查看详情、删除（仅限本人） | 高 |
| R6 | 奶油风 UI：奶茶背景 #F6F5F2、白卡 #FFFFFF、巧克力褐 #2D2722、18px 大圆角 | 高 |

### 2.2 非功能需求

- **性能**：历史查询使用 user_id 索引，单用户记录 < 1000 条
- **安全**：密码永不明文存储，SQL 参数化查询防注入
- **兼容性**：现有 job_positions 表不受影响，新增表幂等建表
- **维护性**：模块化分层，高内聚低耦合

## 三、技术架构

### 3.1 总体架构

采用**模块化分层架构**：

```
认证层（auth_db.py） → 历史持久层（chat_history.py） → 会话管理层（session_manager.py） → 表示层（app.py）
```

### 3.2 模块依赖图

```
app.py
├── utils/auth_db.py          ← 用户注册、登录验证、密码哈希
├── utils/chat_history.py     ← 面试/灯塔记录 CRUD
├── utils/session_manager.py  ← 登录态管理、注销清理

```

### 3.3 数据库设计

在现有 `data/hireinsight.db` 中新增三张表：

#### users 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | TEXT | PRIMARY KEY | UUID |
| username | TEXT | UNIQUE NOT NULL | 用户名（小写） |
| password_hash | TEXT | NOT NULL | PBKDF2-SHA256 哈希 |
| salt | TEXT | NOT NULL | 16 字节随机 Salt（hex） |
| display_name | TEXT | | 显示名称 |
| created_at | TEXT | NOT NULL | 创建时间 |

索引：`idx_users_username` (username)

#### interview_history 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | TEXT | PRIMARY KEY | 记录 ID |
| user_id | TEXT | NOT NULL | 外键 → users.id |
| target_position | TEXT | NOT NULL | 目标岗位 |
| target_company | TEXT | | 目标公司 |
| user_resume | TEXT | | 简历文本 |
| market_report | TEXT | | 市场趋势报告 |

| interview_questions | TEXT | | 面试题列表（JSON） |
| created_at | TEXT | NOT NULL | 创建时间 |

索引：`idx_interview_user_id` (user_id, created_at DESC)

#### lighthouse_history 表

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | TEXT | PRIMARY KEY | 记录 ID |
| user_id | TEXT | NOT NULL | 外键 → users.id |

| created_at | TEXT | NOT NULL | 创建时间 |

索引：`idx_lighthouse_user_id` (user_id, created_at DESC)

### 3.4 登录流程

```mermaid
flowchart TD
    A[启动 app.py] --> B{已登录?}
    B -->|否| C[显示登录/注册页]
    C --> D{选择操作}
    D -->|登录| E[验证用户名密码]
    D -->|注册| F[创建新账户]
    E -->|成功| G[写入 session_state]
    E -->|失败| C
    F -->|成功| G
    F -->|失败| C
    G --> H[渲染主应用]
    B -->|是| H
    H --> I{点击注销?}
    I -->|是| J[logout_user(): 清除 session_state + st.rerun()]
    J --> C
```

### 3.5 历史记录保存时机

- **面试模拟**：在 `run_interview_workflow` 返回结果后，提取 market_report、gap_analysis、interview_questions 调用 `save_interview_record()`
- **灯塔计划**：在 `assessment_node` 返回结果后，从 `st.session_state.lighthouse_results` 提取 tech_tendency、roadmap、user_answers 调用 `save_lighthouse_record()`

## 四、模块详细设计

### 4.1 utils/auth_db.py

职责：用户认证数据库层

核心接口：

```python
def init_auth_tables(db_path: str) -> None
    """幂等建表：users 表"""

def register_user(username: str, password: str, display_name: str, db_path: str) -> tuple[bool, str]
    """注册用户，返回 (是否成功, 提示信息)"""

def verify_login(username: str, password: str, db_path: str) -> dict | None
    """验证登录，成功返回用户信息字典，失败返回 None"""

def get_user_by_id(user_id: str, db_path: str) -> dict | None
    """按 ID 查询用户信息"""
```

安全：
- 密码使用 PBKDF2-SHA256 + 随机 Salt，100000 次迭代
- 所有 SQL 参数化查询
- 用户名强制小写，3-32 位，仅允许字母数字下划线
- 密码至少 6 位

```python
def init_history_tables(db_path: str) -> None
```

### 4.3 utils/session_manager.py

职责：Streamlit session_state 登录态管理

核心接口：

```python
def init_auth(db_path: str) -> None
    """初始化认证表"""

def init_history(db_path: str) -> None
    """初始化历史记录表"""

def is_logged_in() -> bool
    """检查是否已登录"""

def get_current_user() -> dict | None
    """获取当前用户信息"""

def login_user(user_info: dict) -> None
    """写入登录态到 session_state"""

def logout_user() -> None
    """清除登录态和所有页面相关 session_state，末尾强制 st.rerun()
       确保页面瞬间跳转回登录墙，无卡顿感"""
```

## 六、测试建议

1. **注册测试**：输入不同用户名/密码，验证边界条件（短用户名、重复用户名、密码不一致）
2. **登录测试**：正确/错误密码，验证登录态持久化
3. **历史隔离测试**：创建两个用户，各自进行面试模拟和灯塔测评，确认只能查看自己的记录
4. **历史管理测试**：查看列表、查看详情、删除记录，确认删除后不可访问
5. **UI 测试**：检查所有页面配色、圆角、按钮悬停效果是否符合奶油风要求

## 七、风险与注意事项

1. **Streamlit CSS 全局性**：CSS 注入会影响所有页面，确保看板和侧边栏不添加 hover 效果
2. **历史记录重复保存**：灯塔计划使用 `st.session_state["lighthouse_saved"]` 标记避免重复保存
3. **session_state 清理**：`logout_user()` 末尾强制调用 `st.rerun()`，确保注销后页面瞬间跳转回登录墙，避免用户需要再点一次按钮才会刷新的卡顿感。
4. **数据库兼容性**：新增表使用 `CREATE TABLE IF NOT EXISTS`，不影响现有 job_positions 表
